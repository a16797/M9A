# uv and Agent Runtime Migration Plan

This document records the runtime boundaries and maintenance rules after migrating M9A's development environment to `uv`.

## Decision

M9A can use `uv` for development startup, but release packages should not depend on `uv`. The recommended boundary is:

- Use `uv sync` / `uv run` for the development Python environment and tools.
- The source-tree `assets/interface.json` may launch the Agent through `uv run`.
- Release packages should continue to launch the Agent with embedded Python or system Python, without requiring users to install `uv`.
- Keep `requirements.txt` as the release dependency contract; use `pyproject.toml` and `uv.lock` for development and CI locking.
- Packaging scripts must keep rewriting `agent.child_exec` and `agent.child_args` for release packages so the source-tree `uv` command never leaks into release artifacts.

## Current Facts

The current source-tree Agent configuration is in `assets/interface.json`:

```json
"agent": {
    "child_exec": "uv",
    "child_args": [
        "run",
        "--frozen",
        "python",
        "-u",
        "./../agent/bootstrap.py"
    ]
}
```

Release packaging scripts already rewrite this configuration:

- Windows: `./python/python.exe`
- macOS: `./python/bin/python3`
- Linux: `python3`
- Arguments: `["-u", "./agent/bootstrap.py"]`

This means the source tree may move to `uv run`, but release rewrite logic must stay in place.

The current Python version boundary stays aligned:

- ruff/pyright in `pyproject.toml` target Python 3.12.
- CI uses Python 3.12.
- `tools/ci/setup_embed_python.py` uses embedded Python 3.12.10.

This plan does not include upgrading to Python 3.13.

## Reference Boundaries

[Windsland52/create-maa-project](https://github.com/Windsland52/create-maa-project) has a clean two-state Agent template:

- Development command defaults to `uv run python agent/bootstrap.py`.
- `pyproject.toml` / `uv.lock` manage the development environment.
- Release builds rewrite the Agent startup command to platform Python.
- Runtime sync still keeps `requirements.txt` and installs dependencies into the target Python with pip or uv pip.
- `bootstrap.py` uses a requirements sha256 marker to avoid reinstalling dependencies on every startup.

[Windsland52/ArknightsAutoOperator](https://github.com/Windsland52/ArknightsAutoOperator) is a Win32-only project. Its source `interface.json` directly uses:

```json
"agent": {
    "child_exec": "uv",
    "child_args": [
        "run",
        "python",
        "-u",
        "-m",
        "custom.agent"
    ]
}
```

It shows that a Win-only development project can let MaaFramework launch `uv run` directly. M9A, however, has release packages, embedded Python, multiplatform dependencies, and a no-setup expectation for users. Therefore AOO is useful as a development startup reference, not as a release-package model.

## Target Shape

### Python Project Metadata

Extend the current `pyproject.toml` from tool-only configuration to full project metadata:

```toml
[project]
name = "m9a"
requires-python = ">=3.12,<3.13"
dependencies = [
    "maafw==v5.10.4",
    "loguru",
    "pillow",
    "pytz",
    "requests",
]

[dependency-groups]
dev = [
    "black",
    "ruff",
    "pytest",
    "pyright",
    "jsonschema==4.26.0",
    "referencing==0.37.0",
]

[tool.uv]
package = false
```

Also add:

- `.python-version`: pinned to `3.12`
- `uv.lock`: generated by `uv lock` and committed

### Dependency File Responsibilities

Keep `requirements.txt`; it remains the release and bootstrap runtime dependency contract.

| File                   | Responsibility                                                                |
| ---------------------- | ----------------------------------------------------------------------------- |
| `pyproject.toml`       | Development environment, tool config, dependency declaration                  |
| `uv.lock`              | Reproducible development and CI lock                                          |
| `requirements.txt`     | Release dependency list and bootstrap install input                           |
| `requirements-dev.txt` | Keep during the first phase; remove or demote only after CI fully moves to uv |

`[tool.uv] package = false` means M9A is not installed as a package during `uv sync`; the repository is used as an application runtime tree. This avoids adding wheel build configuration just for the uv migration.

### Source-Tree Agent Startup

The source tree may use:

```json
"agent": {
    "child_exec": "uv",
    "child_args": [
        "run",
        "--frozen",
        "python",
        "-u",
        "./../agent/bootstrap.py"
    ]
}
```

`--frozen` ensures startup does not implicitly modify `uv.lock`. If this hurts first-run developer experience, keep `--frozen` only in CI and scripts and omit it from Maa source startup, but document that trade-off in the PR.

### Release Agent Startup

Release packages should still be rewritten to platform Python:

```json
"agent": {
    "child_exec": "./python/python.exe",
    "child_args": [
        "-u",
        "./agent/bootstrap.py"
    ]
}
```

macOS and Linux continue to use `./python/bin/python3` and `python3`. Release packages should not require `.venv`, `uv.lock`, `.python-version`, or a user-installed `uv`.

## Implementation Status

### Phase 1: Move Development Environment to uv

Status: implemented. Developers can run the current Python checks with `uv sync` and `uv run`.

Scope:

- Add `[project]`, `[dependency-groups] dev`, `.python-version`, and `uv.lock`.
- Keep `requirements.txt` and `requirements-dev.txt`.
- Move Python scripts in `package.json` to `uv run --frozen`. On Windows, pnpm may leave residual processes when wrapping `uv.exe` directly, so the scripts call uv through `tools/run-uv.mjs`:

```json
{
    "install:py": "node tools/run-uv.mjs sync --frozen --group dev",
    "format:py": "node tools/run-uv.mjs run --frozen black .",
    "lint:py": "node tools/run-uv.mjs run --frozen ruff check .",
    "typecheck": "node tools/run-uv.mjs run --frozen pyright",
    "test": "node tools/run-uv.mjs run --frozen pytest"
}
```

- Use a full-version `astral-sh/setup-uv` tag in GitHub Actions while keeping Python 3.12.

Acceptance commands:

```bash
uv sync --frozen --group dev
pnpm lint:py
pnpm typecheck
pnpm test
python tools/validate_schema.py --schema-dir deps/tools --resource-dirs assets/resource --exclude-dirs assets/resource/announcement assets/resource/data assets/resource/tasks --interface-files assets/interface.json --task-dirs assets/resource/tasks
```

### Phase 2: Launch Source Agent with uv

Status: implemented. MaaFramework launches the development Agent through `uv run`.

Scope:

- Change the source `assets/interface.json` `agent` block.
- Keep the release rewrite logic in `tools/ci/install.py`, `tools/ci/install_cli.py`, and `tools/ci/install_mxu.py`.
- Add a release rewrite smoke check to prove release artifacts do not contain the `uv` startup command.

Acceptance command:

```bash
python tools/ci/install.py v0.0.0 win-x64
python - <<'PY'
import json

with open("install/interface.json", encoding="utf-8") as file:
    print(json.load(file)["agent"])
PY
```

Windows PowerShell:

```powershell
@'
import json

with open("install/interface.json", encoding="utf-8") as file:
    print(json.load(file)["agent"])
'@ | python -
```

Expected result: `child_exec` is release Python, not `uv`.

### Phase 3: Optimize bootstrap

Status: implemented. The Agent uses a `requirements.txt` sha256 marker to skip repeated pip installation.

Scope:

- Compute the sha256 of `requirements.txt`.
- After successful install, write a marker such as `debug/.m9a-requirements.sha256` or a marker inside the venv.
- On the next startup, skip installation when the marker matches and `maafw` is installed.
- Keep the current install order: local wheels first, configured package indexes as fallback.

Acceptance:

- First startup installs dependencies and writes the marker.
- Second startup does not reinstall dependencies.
- Changing `requirements.txt` triggers reinstall.
- Release and development behavior remain consistent.

### Phase 4: Align the Agent Structure and VSCode Debug Entry

Status: implemented. The project structure now follows the `create-maa-project` Agent template more closely: `bootstrap.py` prepares the runtime, while `main.py` only starts AgentServer.

Scope:

- Source-tree `assets/interface.json` launches `uv run --frozen python -u ./../agent/bootstrap.py`.
- Release packaging rewrites the Agent to platform Python + `./agent/bootstrap.py`.
- `agent/bootstrap.py` prepares the environment, checks dependencies, switches the development work root to `assets/`, and passes runtime state to `main.py`.
- `agent/main.py` stays as a thin entrypoint that normalizes cwd/sys.path and calls `agent_runtime.run_agent()`.
- `maatools.config.mts` maps the VSCode Maa Support uv agent to a debug session.
- `.vscode/launch.json` and `.vscode/tasks.json` provide VSCode tasks for `install python env`, `check`, and `validate schema`.

Acceptance:

```bash
npm run install:py
npm run check
node tools/run-uv.mjs run --frozen python tools/validate_schema.py --schema-dir deps/tools --resource-dirs assets/resource --exclude-dirs assets/resource/announcement assets/resource/data assets/resource/tasks --interface-files assets/interface.json --task-dirs assets/resource/tasks
```

## Non-Goals

- Do not replace `download_deps.py` with `uv`; it handles cross-platform wheel downloads, where explicit pip flags are clearer.
- Do not remove `requirements.txt`.
- Do not delete `requirements-dev.txt` in the first migration pass.
- Do not upgrade to Python 3.13.
- Do not introduce `maa-project.json` / `maa-project.lock.json`.
- Do not change Pipeline business logic, resource paths, or task entries.

## Risks and Mitigations

| Risk                                                                 | Mitigation                                                                                                              |
| -------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------- |
| Users do not have `uv` when running a release package                | Release `interface.json` must be rewritten to platform Python                                                           |
| CI and local Python versions drift                                   | Keep `.python-version`, CI Python, pyright, and ruff targets on 3.12                                                    |
| `uv.lock` and `requirements.txt` drift                               | Document the two-file responsibility; runtime dependency changes must update both                                       |
| First Maa startup fails because `uv run --frozen` has no environment | Require `uv sync --frozen --group dev` first; omit `--frozen` from source Maa startup if needed                         |
| Release artifacts accidentally include development files             | Smoke check that `.venv`, `.python-version`, `uv.lock`, `package.json`, and similar files do not enter the runtime root |
